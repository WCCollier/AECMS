Deploying to Google Cloud Run

Production-grade, serverless containers. Automated via GitHub Actions.

Estimated time: 60–90 minutes  ·  Cost: ~$10–12/month at low traffic

ℹ️ Who this is for

Cloud Run is best suited for owners who are comfortable with cloud infrastructure or want a scalable foundation that grows with their site. The initial setup has more steps than Railway or Coolify, but the deployment pipeline is fully automated after that.

Architecture Overview

  • Cloud Run — runs backend and frontend containers (scales to zero when idle)
  • Cloud SQL — managed PostgreSQL 15 ($7–9/month for smallest instance)
  • Upstash Redis — serverless Redis (free tier is sufficient for low traffic)
  • Google Cloud Storage — two buckets (media + digital), created automatically
  • Secret Manager — stores database URL, JWT secret, Redis URL, and the Settings Encryption Key
  • GitHub Actions — builds Docker images and deploys on every push to the deploy branch

Before You Start

  • A GitHub account with the AECMS repository forked
  • A Google Cloud account with billing enabled
  • The Google Cloud CLI (gcloud) installed on your computer
  • A domain name with access to its DNS settings

Step 1 — Create a GCP Project

  1. Go to console.cloud.google.com → click the project dropdown → New Project.
  2. Enter a project name (e.g. my-aecms-site). Note the auto-generated Project ID (e.g. my-aecms-site-123456) — you'll need it throughout.
  3. Note the Project Number (numeric) shown in the Dashboard.

Step 2 — Enable Required APIs

gcloud config set project YOUR_PROJECT_ID

gcloud services enable \
  run.googleapis.com \
  sqladmin.googleapis.com \
  secretmanager.googleapis.com \
  artifactregistry.googleapis.com \
  storage.googleapis.com \
  cloudresourcemanager.googleapis.com

Step 3 — Create Artifact Registry Repository

gcloud artifacts repositories create aecms \
  --repository-format=docker \
  --location=us-central1 \
  --description="AECMS container images"

Step 4 — Create Cloud SQL Database

# Create the instance (takes 3–5 minutes)
gcloud sql instances create aecms-db \
  --database-version=POSTGRES_15 \
  --tier=db-f1-micro \
  --region=us-central1 \
  --storage-auto-increase

# Create the database and user
gcloud sql databases create aecms --instance=aecms-db
gcloud sql users create aecms --instance=aecms-db --password=YOUR_DB_PASSWORD

Step 5 — Create Upstash Redis

  1. Go to upstash.com → create a free account.
  2. Create a Redis database: choose the region closest to us-central1.
  3. Copy the Redis URL from the database details page.

Step 6 — Store Secrets in Secret Manager

# Generate a JWT secret
JWT_SECRET=$(openssl rand -hex 64)

# Generate the Settings Encryption Key — SAVE THIS SOMEWHERE SAFE
SEK=$(openssl rand -hex 32)

# Store each secret
echo -n "postgresql://aecms:YOUR_DB_PASSWORD@/aecms?host=/cloudsql/YOUR_PROJECT_ID:us-central1:aecms-db" \
  | gcloud secrets create aecms-database-url --data-file=-

echo -n "$JWT_SECRET" | gcloud secrets create aecms-jwt-secret --data-file=-

echo -n "YOUR_UPSTASH_REDIS_URL" | gcloud secrets create aecms-redis-url --data-file=-

echo -n "$SEK" | gcloud secrets create aecms-sek --data-file=-

echo "⚠️  Save your SEK: $SEK"
⚠️ Save your Settings Encryption Key

The SEK value printed above is your Settings Encryption Key. It is stored in Secret Manager, but you should also write it down and keep it in a password manager. If Secret Manager is deleted or the secret is rotated without backing it up, all credentials stored in Admin Settings become permanently unreadable.

Step 7 — Create a Service Account

gcloud iam service-accounts create aecms-backend \
  --display-name="AECMS Backend Service Account"

SA="aecms-backend@YOUR_PROJECT_ID.iam.gserviceaccount.com"

# Grant required roles
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID --member="serviceAccount:$SA" --role="roles/cloudsql.client"
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID --member="serviceAccount:$SA" --role="roles/secretmanager.secretAccessor"
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID --member="serviceAccount:$SA" --role="roles/storage.admin"
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID --member="serviceAccount:$SA" --role="roles/run.admin"
gcloud projects add-iam-policy-binding YOUR_PROJECT_ID --member="serviceAccount:$SA" --role="roles/artifactregistry.writer"

# Create a key for GitHub Actions
gcloud iam service-accounts keys create gcp-sa-key.json --iam-account="$SA"

Step 8 — Set GitHub Variables and Secrets

In your forked GitHub repository, go to Settings → Secrets and variables → Actions.

Add these Variables (not secrets):

Variable nameValue
GCP_PROJECT_NUMBERYour numeric project number
GCP_PROJECT_IDYour text project ID
APP_DOMAINYour domain (e.g. myblog.com)
GCS_MEDIA_BUCKETe.g. myblog-media
GCS_DIGITAL_BUCKETe.g. myblog-digital
SETTINGS_KMS_SECRET_IDaecms-sek

Add this Secret:

Secret nameValue
GCP_SA_KEYPaste the entire contents of gcp-sa-key.json

Then delete gcp-sa-key.json from your computer — it's now in GitHub Secrets.

Step 9 — Deploy

  1. In your forked repo, go to the Actions tab → select the Deploy to Cloud Run workflow → click Run workflow.
  2. The workflow: runs tests → creates GCS buckets (idempotent) → builds and pushes Docker images → deploys backend → deploys frontend.
  3. After ~6 minutes, the workflow prints the backend and frontend Cloud Run URLs.
  4. Visit the frontend URL to see your site or the setup wizard.

Step 10 — Map Your Domain

  1. Map your domain to Cloud Run: 
    gcloud beta run domain-mappings create \
  --service aecms-frontend \
  --domain yourdomain.com \
  --region us-central1

gcloud beta run domain-mappings create \
  --service aecms-frontend \
  --domain www.yourdomain.com \
  --region us-central1
  2. Run gcloud beta run domain-mappings describe --domain yourdomain.com --region us-central1 to get the DNS records you need to add.
  3. In your domain registrar's DNS, add the A/AAAA and CNAME records shown.
  4. DNS propagation takes up to 24 hours. Cloud Run handles SSL automatically once DNS is active.
✅ Set APP_DOMAIN before first deploy

Set APP_DOMAIN to your final domain name (e.g. myblog.com) in GitHub Variables before running the first deploy. The workflow bakes this into the container as APP_URL. If you change it later, you will need to redeploy.
