Deploying to Railway

The easiest path: no command line required.

Estimated time: 30–45 minutes  ·  Cost: ~$5/month hobby plan

✅ What Railway provides

Railway manages your containers, builds from GitHub, and hosts managed PostgreSQL and Redis — all in one dashboard. You do not need to configure servers or install Docker yourself.

Before You Start

  • A GitHub account with the AECMS repository forked to your account
  • A Railway account (sign up with GitHub — free)
  • A file storage provider if you want cloud storage: Cloudflare R2 (10 GB free) is recommended
  • An SMTP provider: Resend (3,000 emails/month free) is recommended

Step 1 — Fork the Repository

Go to the AECMS repository on GitHub and click Fork. This creates your own copy that Railway will deploy from.

ℹ️ Why fork?

Forking gives you your own copy of the code. Railway deploys from your fork, so you can update AECMS by pulling changes from the original repo into your fork and redeploying.

Step 2 — Create a Railway Project

  1. Log in to Railway and click New Project.
  2. Choose "Deploy from GitHub repo" and select your forked AECMS repository.
  3. Railway will detect the Dockerfile and offer to deploy. Click "Add service" but do not deploy yet — you need to add the database and Redis first.

Step 3 — Add PostgreSQL

  1. In your Railway project, click New → Database → PostgreSQL.
  2. Railway creates a managed PostgreSQL database. Click it to see the connection details.
  3. Copy the DATABASE_URL value — you will need it in Step 6.
✅ Railway auto-connects services

If you reference ${{Postgres.DATABASE_URL}} in your service's environment variables, Railway automatically injects the database URL. You can use this instead of copying the value manually.

Step 4 — Add Redis

  1. In your Railway project, click New → Database → Redis.
  2. Railway creates a managed Redis instance.
  3. Copy the REDIS_URL value (or use ${{Redis.REDIS_URL}} in env vars).

Step 5 — Configure the Backend Service

Click on your AECMS service (the one from GitHub) → Variables. Add each of the following:

NODE_ENV=production
DATABASE_URL=${{Postgres.DATABASE_URL}}
REDIS_URL=${{Redis.REDIS_URL}}

# Generate these with: openssl rand -hex 64
JWT_SECRET=YOUR_LONG_RANDOM_STRING
# Generate with: openssl rand -hex 32
SETTINGS_ENCRYPTION_KEY=YOUR_32_BYTE_HEX_KEY

JWT_EXPIRATION=15m
REFRESH_TOKEN_EXPIRATION=7d
SETTINGS_KMS_PROVIDER=local
EMAIL_PROVIDER_TYPE=smtp

# Set to your Railway app URL for now; update to your custom domain later
APP_URL=https://YOUR-APP.up.railway.app
FRONTEND_URL=https://YOUR-APP.up.railway.app
API_URL=https://YOUR-APP.up.railway.app
FRONTEND_ADMIN_URL=https://YOUR-APP.up.railway.app/admin

# Storage — start with local, configure cloud storage later in Admin Settings
STORAGE_PROVIDER_TYPE=local
STORAGE_PATH=/app/uploads
⚠️ Generate real random values

Do not use placeholder text for JWT_SECRET or SETTINGS_ENCRYPTION_KEY. Run these commands in a terminal to generate secure values:

openssl rand -hex 64   # for JWT_SECRET
openssl rand -hex 32   # for SETTINGS_ENCRYPTION_KEY

Store SETTINGS_ENCRYPTION_KEY somewhere safe. Losing it makes all secrets stored in Admin Settings permanently unreadable.

Step 6 — Deploy

  1. Click Deploy on the AECMS service. Railway will build the Docker image from your fork and start the container.
  2. Watch the build logs. A successful deploy ends with NestJS application listening on port 4000.
  3. Railway assigns a URL like your-project.up.railway.app. Open it — you should see the AECMS site or the setup wizard.
ℹ️ Frontend and backend

AECMS runs the frontend (Next.js) and backend (NestJS) as separate services. If Railway deploys only one, add a second service from the same repo and set its root directory to /frontend. Set the frontend's BACKEND_URL variable to the backend's Railway URL.

Step 7 — Set Up File Storage (Optional)

Local storage works on Railway if you add a persistent volume. For cloud storage with Cloudflare R2:

  1. Create a Cloudflare account and go to R2 Object Storage.
  2. Create two buckets: my-site-media (public) and my-site-digital (private).
  3. Go to R2 → Manage R2 API Tokens → Create Token with "Object Read & Write" permission.
  4. In Railway, add these variables to your backend service:
    STORAGE_PROVIDER_TYPE=s3
    S3_ENDPOINT=https://ACCOUNT_ID.r2.cloudflarestorage.com
    S3_REGION=auto
    Then add bucket names and credentials via Admin Settings → File Storage after the wizard runs.

Step 8 — Map Your Custom Domain

Railway's default URL works immediately. To use your own domain:

  1. In Railway, open your service → Settings → Networking → Custom Domain.
  2. Enter your domain (e.g. myblog.com). Railway shows you a CNAME record to add.
  3. Go to your domain registrar's DNS settings and add the CNAME record pointing to Railway's provided target.
  4. DNS propagation takes 15 minutes to 24 hours. Railway provides SSL automatically via Let's Encrypt once DNS is active.
  5. Once DNS is live, update APP_URL, FRONTEND_URL, API_URL, and FRONTEND_ADMIN_URL to your custom domain and redeploy.
✅ Run the wizard now, update URL later

You can run the setup wizard on the *.railway.app URL while DNS propagates. Just remember to update the APP_URL environment variables to your final domain and redeploy once DNS is live.
